# 翁家翌：工程哲学与 Infra 方法论（Engineering）

## 一句话定位
- 不是“谁 idea 更天才”，而是“谁验证更快、更稳、更可复现”。

## 核心主张
- 重工程，轻炼丹。
- idea is cheap，验证效率和验证质量是稀缺资源。
- 模型性能差距往往来自基础设施 bug 清理能力，而非单次调参运气。

## 工程第一性原理
1. 迭代速度优先：单位时间内可完成多少次有效验证，决定研发上限。
2. 一致性优先：接口、数据流、模块边界必须自洽，否则技术债指数增长。
3. 可观测性优先：没有测量就没有优化。
4. 可复现优先：能稳定复现比单次 SOTA 更关键。

## EnvPool 案例（资料提炼）

### 问题定义
- 传统 Python 多进程方案开销高、上下文切换重，吞吐受限。

### 工程策略
- 用 C++ 线程池替代低效子进程模型。
- 使用异步事件驱动，避免全局被最慢环境拖累。
- 采用预分配内存与无锁队列，降低同步和内存管理开销。

### 结果指标（资料提及）
- Atari 场景达到约 1M FPS。
- MuJoCo 场景达到约 3M FPS。

### 方法论沉淀
- 优化不是“玄学调参”，而是有组织地消除瓶颈与 bug。

## 大模型时代下 RL 工程重心变化
- 传统 toy task：瓶颈常在环境模拟与交互。
- LLM 后训练：瓶颈转移到模型采样与训练吞吐。
- 结论：Infra 目标从“能跑”升级为“能规模化稳定运行”。

## Human-in-the-loop 与 Reward Hacking
- 单一奖励指标会带来 reward hacking 风险。
- 需要高频人机协同评估 checkpoint，避免模型“指标正确、行为失真”。
- 这要求评价系统与训练系统同时工程化，而不是只调模型本体。

## 工程反模式（高风险）
- 无指标优化：不知道在优化什么。
- 盲炼丹：把随机试错当成方法论。
- 代码风格漂移：短期提速，长期崩盘。
- 只看模型分数：忽略稳定性、成本、可维护性。

## 实战检查清单
- 是否定义吞吐、延迟、稳定性、成本四类指标。
- 是否有可复现脚本和版本锁定。
- 是否有性能回退检测和报警机制。
- 是否定期清理技术债并重构关键瓶颈模块。
- 是否区分“可上线优化”和“论文型优化”。

## 高频原话（语义提炼）
- “每家 infrastructure 都有不同程度的 bug，谁修的 bug 多，谁模型性能就越好。”
- “管公司就像管代码库，一旦不一致，身子动了脚没动。”
- “软件工程很重要，tech debt 会把 research 价值吃掉。”

## 新增补充（WhynotTV 第4期口述）
- 早期 RL 体验显示：依赖启发式防崩训练的“炼丹”流程，可复现性差、迁移性弱。
- MILA 阶段暴露出硬约束：方向正确不足以保证结果，算力与工程能力决定可验证上限。
- OpenAI 实践定位为“卖铲子”：通过 Post-training RL Infra 服务研究链路，优先提高全局验证吞吐。
- LLM 时代瓶颈迁移：环境交互可降至微秒级，但模型采样与训练可达百秒到千秒级，系统瓶颈转向算力与并行效率。
- Reward Hacking 防线：checkpoint 质量难靠单一自动指标判定，需要人机协同评测闭环。
- 持续重构原则：面对技术债与外部竞速，关键系统需要阶段性推倒重构，而不只是持续打补丁。
